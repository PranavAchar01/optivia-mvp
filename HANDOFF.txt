1. WHAT OPTIVIA IS
--------------------------------------------------------------------------------

A pre-execution optimisation layer for agentic coding CLIs (Claude Code in
Stage 1). It sits upstream of Claude Code: takes a sloppy human prompt,
classifies it, scores it on 6 dimensions, asks clarifying questions if needed,
synthesises a master prompt, picks the right model tier, plans a sub-agent
fleet, and dispatches everything into Claude Code.

Three-stage roadmap (per architecture spec §1):
  Stage 1 — Wrapper MVP        (DONE — this repo)
  Stage 2 — Custom routing     (logged trace data trains proprietary models)
  Stage 3 — V1 production      (online learning, multi-tier execution, caching)

7. THE DISPATCH FLOW (newest piece — this is what wires everything together)
--------------------------------------------------------------------------------

The desktop app has TWO panels stacked vertically:

  +--------------------------------------------------+
  | OPTIVIA · pre-execution layer       [SETTINGS]   |
  +--------------------------------------------------+
  |  ● ● ●  OPTIVIA · LANGGRAPH ENGINE               |
  |  › <your prompt>                                 |
  |  cache_lookup  miss                              |
  |  fast_intent   NEW_CODE · conf 0.82              |
  |  classify      complexity 0.84 · risk 0.72 · …   |
  |  …                                               |
  |  dispatch      queued · run claude in your term  |  <- top pane (60%)
  +--------------------------------------------------+
  |  ● ● ●  SHELL · TYPE `claude` TO DISPATCH        |
  |  optivia  shell ready — submit a prompt to …    |
  |  $                                               |  <- bottom pane (40%)
  +--------------------------------------------------+
  | [ describe a coding task — press enter to … ] [ENTER ↵] |
  +--------------------------------------------------+

The bottom pane is a REAL PTY running the user's $SHELL with an init script
that defines a `claude()` function. The init script lives at:
  ~/.optivia/shell/init.sh
and is auto-written on first launch by Rust (see desktop/src-tauri/src/pty.rs).

Submit flow:
  1. User types prompt → Enter
  2. Frontend POSTs to {apiBase}/optimize
  3. Backend runs full LangGraph pipeline
  4. On success, frontend invokes Tauri command `dispatch_to_claude(payload)`
  5. Rust writes payload to:
       ~/.optivia/queue/<timestamp>.json   (history)
       ~/.optivia/current.json             (latest, what the shim reads)
  6. LangGraph panel's `dispatch` line flips green: "queued · run claude …"

Consume flow (in the embedded shell):
  1. User types `claude` (no args) + Enter
  2. The aliased `claude` function:
       a. Reads ~/.optivia/current.json
       b. Extracts {prompt, proxy_base, trace_id}
       c. Renames the file → *.consumed.json (so it can't fire twice)
       d. Sets ANTHROPIC_BASE_URL=$proxy_base/proxy/req/$trace_id
       e. Runs `command claude "<the master prompt>"`
  3. Claude Code starts with the optimised prompt as the first message
  4. Every Anthropic API call from that Claude Code session goes through
     Optivia's proxy (backend/proxy.py), which:
       - Injects the system preamble with cache_control: ephemeral
       - Logs every call to the trace
       - (Future Stage 2) forces the chosen model

Plain `claude --resume`, `claude /init`, `claude -p "foo"`, etc. fall through
unchanged — only bare `claude` with a queued prompt triggers injection.

Optional install for the user's regular terminal (not the embedded shell):
  Settings → INSTALL SHIM
  Adds the same shim block (between marker comments) to ~/.zshrc and ~/.bashrc
  so the user can run `claude` in iTerm/Terminal/Ghostty and pick up the
  Optivia queue too. UNINSTALL removes it cleanly using the markers.

8. KEY MATH (architecture spec §4)
--------------------------------------------------------------------------------

Score vector v_t = (κ_t, σ_t, τ_t) with 6 dimensions feeding into κ:
  κ_t = round(1 + 9 · avg(δ_s, δ_a, δ_r, δ_d, δ_c)) ∈ {1,…,10}
  σ_t = 1 − δ_a   (specificity = inverse ambiguity)

Clarification trigger:
  (κ ≥ 7 ∧ σ < 0.5) ∨ (δ_s ≥ 0.7 ∧ c_t < 0.5) ∨ (κ = 10)

Routing (heuristic — Stage 1 active router):
  κ table:        1-3→Haiku · 4-5→Haiku · 6-7→Sonnet · 8→Sonnet · 9-10→Opus
  n_agents table: 1,1,1,2,3,4,5,6,8,8     (more fragmented than spec defaults)
  Hard guardrail: risk ≥ 0.7 never routes to Haiku

Shadow routers (logged but not active):
  routellm_mf_v1   — bilinear win-prob with cascade rule (τ=0.08)
  llm_judge_v1     — Claude Haiku structured-output

Quality scalar:
  Q_t = ¼ · (goal_met + no_errors + matches_conventions + minimal_changes)

9. KNOWN ISSUES / GOTCHAS
--------------------------------------------------------------------------------

[CRITICAL] Embedded shell PATH is missing /opt/homebrew/bin and /usr/local/bin
  Symptom (seen in screenshot): `zsh: command not found: claude`
  Cause: Tauri inherits a minimal PATH; the spawned PTY's $SHELL doesn't load
         the user's full login profile, so npm/claude/brew binaries miss.
  Fix:   In desktop/src-tauri/src/pty.rs, before `cmd.env(...)`, prepend the
         common bin dirs:
           let mut path = std::env::var("PATH").unwrap_or_default();
           for p in ["/opt/homebrew/bin", "/usr/local/bin", "/usr/bin"] {
             if !path.contains(p) { path = format!("{p}:{path}"); }
           }
           cmd.env("PATH", path);
         Better: invoke `$SHELL -l -c env` once at startup to capture the
         user's full login PATH.

[MEDIUM] pgvector for Postgres 18 had to be built from source
  brew install pgvector installs a binary that doesn't link against PG18.
  Workaround used:
    cd /tmp && git clone --depth 1 https://github.com/pgvector/pgvector.git
    cd pgvector
    PG_CONFIG=/opt/homebrew/opt/postgresql@18/bin/pg_config make install
  This needs to be redone on a fresh machine.

[MEDIUM] backend/config.py force-loads .env to override empty env vars
  Claude Code (the desktop tool) sets ANTHROPIC_API_KEY="" in spawned shells,
  which would otherwise shadow the value in .env. The fix is at the top of
  config.py — it manually loads .env and only sets keys when the env var is
  empty/missing. Don't remove this.

[LOW] /traces/{workspace_id} requires a real UUID
  Frontend uses 00000000-0000-0000-0000-000000000000 as the default.
  Backend's POST /optimize defaults workspace_id to the same UUID when omitted
  so traces accumulate under one workspace.

[LOW] execute_via_claude_code node is a SIMULATED stub
  In Stage 1, the LangGraph "execute" node doesn't actually run Claude Code —
  it logs that it would. Real execution happens via the desktop dispatch flow
  (the user runs `claude` themselves). Stage 2 should wire a real subprocess
  spawn here for headless / CI use.

[LOW] Anthropic API key in .env is hot — rotate before any public release
  See line 2 of .env. Created during dev, not for production.

[LOW] Voyage AI free tier is rate-limited (3 RPM, 10K TPM)
  Add a payment method at voyageai.com for production. Embeddings fail open
  (return zero vectors) so the pipeline doesn't crash.

[LOW] frontend/ (Next.js) is now LEGACY
  Replaced by desktop/. Kept for reference only — safe to delete with:
    rm -rf frontend/

[LOW] Existing Optivia.app bundles are unsigned
  macOS will Gatekeeper-block them on first open. Right-click → Open, or
  spctl --add Optivia.app. For real distribution, sign + notarise via Apple
  Developer ID and add the secrets to the GitHub Actions workflow.
